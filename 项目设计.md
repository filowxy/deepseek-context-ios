# deepseek+app — 技术文档

**版本**: 1.13
**状态**: 设计阶段
**范围**: iOS APP 内嵌 DeepSeek WebView 的上下文管理增强系统

---

## 一、产品概述

### 1.1 核心问题

AI 在长对话中容易遗忘早期讨论的重要决策。跨对话时，新对话完全失忆。用户需要反复强调“记住这个”，但 AI 不会主动回想。

### 1.2 解决方案

三个核心机制：**标记、提醒、搜索**。

| 机制 | 作用 | 触发者 |
|------|------|--------|
| 标记 | 将重要信息存入持久化上下文库 | AI 或用户 |
| 提醒 | 按等级定时提醒 AI 不要遗忘 | APP 自动 |
| 搜索 | AI 主动检索历史标记 | AI |

辅助机制：**引用**、**全局上下文**、**对话管理**、**技能系统**、**联网搜索**。

## 三、数据模型

### 3.1 对话表

```sql
CREATE TABLE conversations (
    id TEXT PRIMARY KEY,
    title TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    is_active INTEGER DEFAULT 1
);
```

### 3.2 上下文标记表

```sql
CREATE TABLE context_marks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    mark_id INTEGER NOT NULL,
    conversation_id TEXT NOT NULL REFERENCES conversations(id),
    lev INTEGER NOT NULL CHECK(lev IN (0,1,2,3)),
    type TEXT NOT NULL CHECK(type IN ('userask', 'complex')),
    content TEXT NOT NULL,
    idem_key TEXT UNIQUE,
    created_counter INTEGER NOT NULL,
    last_remind_counter INTEGER,
    created_at TEXT NOT NULL,
    deleted INTEGER DEFAULT 0,
    deleted_at TEXT,
    UNIQUE(conversation_id, mark_id)
);

CREATE INDEX idx_context_marks_reminder ON context_marks(conversation_id, deleted, created_counter);
```

| 字段 | 说明 |
|------|------|
| `idem_key` | 幂等键，格式：`{conversation_id}_{message_index}_{type}_{sequence}` |
| `last_remind_counter` | 上次触发提醒时的对话轮次。NULL 表示从未提醒 |
| `deleted_at` | 删除时间戳，配合回收站功能 |

### 3.3 标签关联表

```sql
CREATE TABLE mark_tags (
    mark_id INTEGER NOT NULL,
    tag TEXT NOT NULL,
    PRIMARY KEY (mark_id, tag),
    FOREIGN KEY (mark_id) REFERENCES context_marks(id) ON DELETE CASCADE
);

CREATE INDEX idx_mark_tags_tag ON mark_tags(tag);
```

### 3.4 全文搜索索引

```sql
CREATE VIRTUAL TABLE context_marks_fts USING fts5(
    content,
    content=context_marks,
    content_rowid=id
);

CREATE TRIGGER context_marks_ai AFTER INSERT ON context_marks BEGIN
    INSERT INTO context_marks_fts(rowid, content) VALUES (new.id, new.content);
END;

CREATE TRIGGER context_marks_ad AFTER DELETE ON context_marks BEGIN
    INSERT INTO context_marks_fts(context_marks_fts, rowid, content) VALUES('delete', old.id, old.content);
END;

CREATE TRIGGER context_marks_au AFTER UPDATE ON context_marks
WHEN old.content IS NOT new.content
BEGIN
    INSERT INTO context_marks_fts(context_marks_fts, rowid, content) VALUES('delete', old.id, old.content);
    INSERT INTO context_marks_fts(rowid, content) VALUES (new.id, new.content);
END;
```

### 3.5 对话链接表

```sql
CREATE TABLE conversation_links (
    conversation_id TEXT PRIMARY KEY REFERENCES conversations(id),
    parent_conversation_id TEXT REFERENCES conversations(id),
    linked_at TEXT NOT NULL
);
```

### 3.6 对话计数表

```sql
CREATE TABLE conversation_counter (
    conversation_id TEXT PRIMARY KEY REFERENCES conversations(id),
    count INTEGER DEFAULT 0
);
```

### 3.7 全局上下文表

```sql
CREATE TABLE global_context (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content TEXT NOT NULL,
    created_at TEXT NOT NULL,
    deleted INTEGER DEFAULT 0
);
```

### 3.8 全局上下文建议日志表

```sql
CREATE TABLE global_suggestion_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    parent_id INTEGER REFERENCES global_suggestion_log(id),
    content TEXT NOT NULL,
    reason TEXT,
    content_hash TEXT,
    status TEXT NOT NULL CHECK(status IN ('accepted', 'rejected')),
    rejection_feedback TEXT,
    created_at TEXT NOT NULL
);
```

## 四、编号规则

| 规则 | 实现方式 |
|------|---------|
| 同一对话内递增 | 插入前查询当前对话最大 `mark_id` + 1 |
| 延续对话继承 | 子对话起始值 = MAX(父对话最大, 子对话当前最大) + 1 |
| 删除不回收 | `mark_id` 删除后不复用 |

## 五、对话管理

### 5.1 对话列表加载

APP 启动时自动加载所有活跃对话，按 `updated_at` 倒序排列。

**活跃对话数量上限：** 系统设定软上限为 50 个活跃对话。当活跃对话数达到 50 时，APP 提示用户归档旧对话。配额仅约束无子对话的对话——已有子对话延续的父对话不计入活跃配额，不计入的原因是此类对话不可归档。

此设计确保：无子对话的普通对话始终占据配额大头，用户可通过归档它们腾出空间；父对话不占配额则避免了归档阻塞问题。

### 5.2 延续上下文

一次性操作，建立后不可更改。子对话继承父对话的标记库、项目技能、对话计数。

### 5.3 对话归档与删除

归档时 `is_active` 设为 0。已有子对话延续的父对话**允许归档**（标记为只读、从活跃列表隐藏），但保留延续关系。子对话的跨对话搜索和继承不受父对话归档影响——归档父对话在搜索时豁免 `is_active` 限制。已有子对话延续的对话不可删除。

## 六、对话计数规则

### 6.1 计数定义

**一轮对话** = 用户点击发送 → AI 完成最终回复气泡渲染（包含中间工具调用序列）。

在此周期结束时计数 +1。工具调用中间步骤和系统注入不计入。

### 6.2 计数触发点

| 事件 | 计数行为 |
|------|---------|
| 用户点击发送按钮 | 标记本轮开始 |
| AI 最终回复气泡渲染完毕 | 计数 +1，本轮结束 |
| 工具调用中间步骤 | 不计入 |
| `re-main` 提醒注入 | 不计入 |
| 系统纠偏提示 | 不计入 |
| `<skill-load>` 注入 | 不计入 |

### 6.3 计数纠偏

用户在设置中可查看当前对话计数并手动修正。提醒逻辑基于 `last_remind_counter` 字段，不受计数修正影响。

### 6.4 延续对话计数初始化

新对话建立延续关系时，计数器初始值 = 父对话的 `count`。

## 七、通信协议：AI 输出的 XML 标签

### 7.1 消息完整性检测与思考过程过滤

注入脚本仅解析 AI "最终回复"部分的 XML 标签，忽略思考过程。

**渲染稳定性检测：** 消息气泡在 DOM 中出现后，其内部 DOM 在 200ms 内无任何变化（`MutationObserver` 无新记录），视为渲染完毕。

每轮对话开始时，APP 向 AI 注入当前轮次编号：

```
<system>当前对话轮次: 5, 本轮标记序号从 1 开始</system>
```

### 7.2 标记用户原话

```xml
<main>
{"type": "userask", "lev": 0, "tags": ["洞穴"], "idem_key": "conv8_5_userask_1", "content": "洞穴系统必须兼容原版生物群系标签"}
</main>
```

### 7.3 标记 AI 总结

```xml
<main>
{"type": "complex", "content": "结构生成走 TerraBlender 路线", "lev": 1, "tags": ["结构"], "idem_key": "conv8_5_complex_2"}
</main>
```

### 7.4 幂等键

格式：`{conversation_id}_{message_index}_{type}_{sequence}`。

| 组成部分 | 来源 | 示例 |
|---------|------|------|
| `conversation_id` | 当前对话标识 | `conv8` |
| `message_index` | APP 注入的当前轮次编号 | `5` |
| `type` | `userask` 或 `complex` | `userask` |
| `sequence` | 本轮内标记序号（从 1 递增） | `1` |

UPSERT 时更新 `content`、`tags`，`deleted` 重置为 0。`lev` 在同一轮内允许修正，跨轮修改须删除旧标记再新建。

### 7.5 删除标记

```xml
<de-main>
{"id": 3}
</de-main>
```

### 7.6 恢复标记

```xml
<recover-mark>
{"id": 3}
</recover-mark>
```

### 7.7 主动召回

```xml
<recall-context>
{"query": {"search": "TerraBlender", "scope": "all"}}
</recall-context>
```

`scope`：`current` / `parent` / `all`。搜索深度默认 3 层。所有查询必须包含 `WHERE cm.deleted = 0`。

### 7.8 召回排序管道

| 阶段 | 规则 |
|------|------|
| 1 | FTS5 全文搜索命中，按 BM25 相关度排序 |
| 2 | 标签精确匹配命中但未出现在阶段 1 中的，按 `created_at` 倒序追加 |
| 3 | 最终截取前 N 条 |

### 7.9 召回结果限制

默认 7 条（可调 1-20）。超出时附带总数提示和唯一 `search_id`。

**召回结果完整 JSON 结构：**

```json
{
    "items": [
        {
            "mark_id": 1,
            "lev": 1,
            "content": "洞穴系统必须兼容原版生物群系标签",
            "tags": ["洞穴"],
            "created_at": "2026-06-27T14:30:00Z"
        }
    ],
    "total": 12,
    "truncated": true,
    "search_id": "recall-co3-tkkx3se",
    "message": "找到 12 条相关标记，以下是匹配度最高的 7 条..."
}
```

`search_id` 存储在 APP 内存中，进程存活期间有效，重启失效。AI 从 `search_id` 字段获取值，用于构造 `<all>` 请求。

### 7.10 请求查看全部

```xml
<all>
{"type": "context", "searchinfo": "recall-co3-tkkx3se"}
</all>
```

### 7.11 搜索网页

```xml
<search>
{"q": "Minecraft NeoForge cave generation", "depth": "detailed"}
</search>
```

| 参数 | 说明 |
|------|------|
| `q` | 搜索提示词 |
| `depth` | `quick`（前 3 条）、`normal`（前 7 条，默认）、`detailed`（前 10 条） |

搜索引擎接入规格：接口为 Bing Web Search API v7，密钥存储在 iOS Keychain，未配置时回退 DuckDuckGo。超时 10 秒。

**搜索返回 JSON 结构：**

```json
{
    "query": "Minecraft NeoForge cave generation",
    "total_estimated": 1200000,
    "results": [
        {
            "title": "Minecraft Forum - Custom Cave Generation",
            "url": "https://www.minecraftforum.net/...",
            "snippet": "This tutorial explains how to create custom cave generators..."
        }
    ]
}
```

| 字段 | 说明 |
|------|------|
| `title` | 网页标题 |
| `url` | 网页链接 |
| `snippet` | 搜索结果摘要 |

不同深度对应条目数：`quick` 返回 3 条，`normal` 返回 7 条，`detailed` 返回 10 条。无结果时 `results` 为空数组。

### 7.12 浏览网页

```xml
<open>
{"type": "url", "dis": "https://github.com/MinecraftForge/ForgeGradle"}
</open>
```

网页抓取规格：`URLSession`，`User-Agent: WorldScapeApp/1.0`，超时 15 秒，最大体积 2 MB。优先 readability 算法，失败时返回纯文本前 4000 字符。遵守 `robots.txt`。

**返回摘要格式：**

```
标题: {page_title}
{正文提取内容}
```

标题取自 `<title>` 标签，无标题时该行省略。

### 7.13 AI 调用 Skill

```xml
<call-skill>name:'审查输出'</call-skill>
```

此标签必须在 AI 的独立首轮输出中出现——该轮回复仅包含此标签，无其他内容。APP 拦截后：

1. 暂存用户原输入及当前对话上下文（包含本轮之前的全部消息）
2. 将 Skill 的 `description` 作为系统消息注入
3. 重新发起请求：系统消息 + 暂存的用户原输入 + 完整上下文

此重放机制确保用户附件、上文引用等不丢失。

### 7.14 工具调用并发控制

当 AI 在同一轮回复中输出多个 `<search>` 或 `<open>` 时，APP 的全局限制：

| 工具类型 | 最大并发数 | 超出处理 |
|---------|----------|---------|
| `<search>` | 1 | 排队，按输出顺序执行 |
| `<open>` | 1 | 排队，按输出顺序执行 |

总等待超时 30 秒。超时后未执行的排队的调用被丢弃，并注入系统提示通知 AI 哪些调用未执行。

## 八、通信协议：APP 自动注入

### 8.1 被动提醒

基于 `last_remind_counter` 计算提醒周期。首次提醒在第 5 轮，之后按等级间隔。

| 等级 | 首次提醒 | 后续周期 |
|------|---------|---------|
| 0 | 第 5 轮 | 每 10 轮 |
| 1 | 第 5 轮 | 每 20 轮 |
| 2 | 第 5 轮 | 每 30 轮 |
| 3 | 第 5 轮 | 每 50 轮 |

### 8.2 全局上下文提示

新对话开始时弹出确认卡片。10 秒无操作自动注入。

### 8.3 AI 主动建议全局上下文

```xml
<global-suggest>
{"content": "World Scape 高度上限 380 格", "reason": "影响所有地形生成的核心约束"}
</global-suggest>
```

### 8.4 引用注入

长按 AI 回复 → 选择文本 → 确认引用，以原文形式注入。

### 8.5 技能列表注入

```
<skill-load>
count:'1',name:'设定目标',when-to-use:'用户需要制定修复计划时'
count:'2',name:'审查输出',when-to-use:'用户要求审查代码时'
</skill-load>
```

若可用 Skill 超过 10 个，仅注入前 10 个。**排序规则：** 最近使用的 Skill 优先，其次按最近修改时间倒序，最后按名称字母序。AI 可通过 `<call-skill>` 动态获取未在列表中的 Skill。

### 8.6 用户调用 Skill

用户选择 Skill 后：输入框为空时拒绝发送并提示「请输入内容」；非空时 APP 将 `description` 作为系统消息注入。`<skill>` 标签仅内部通信。

## 十、WebView DOM 脆弱性应对

### 10.1 健康检测

注入脚本每次操作前检测关键 DOM 节点，连续 3 次未找到则报告错误。

### 10.2 降级模式

**触发条件：** 最近 30 次操作中，失败次数 ≥ 10 次时触发降级提示。

**纯净模式：** 关闭所有 JS 注入增强，仅保留基础 WebView 对话和 Native 层服务。用户可随时手动退出。

## 十一、层级协作

| 层级 | 注入时机 | 作用 |
|------|---------|------|
| **Skill** | 对话开始时 + 手动调用 + AI 主动调用 | 怎么回应 |
| **重要上下文** | 每条发言前检查提醒 | 记住什么 |
| **全局上下文** | 新对话开始时，10 秒确认 | 了解用户 |
| **MCP** | AI 输出工具调用 XML 时 | 能操作什么 |
| **搜索/浏览** | AI 输出 `<search>` / `<open>` 时 | 知道什么 |

---

## 十二、错误处理

| 场景 | APP 行为 |
|------|---------|
| `lev` 越界 / `type` 非法 / `content` 为空 | 拒绝创建 |
| 删除/恢复不存在的编号 | 返回错误 |
| `searchinfo` 无效 | 返回错误 |
| 搜索无结果 | 返回空数组 |
| 父对话未找到或已归档 | 回退或跳过（归档父对话被延续时豁免限制） |
| JSON 解析失败 | 注入系统提示，通知 AI 重试 |
| 健康检测滑动窗口失败率超阈值 | 提示切换纯净模式 |
| 搜索 API 超时 | 返回超时错误 |
| 网页抓取超时或超限 | 返回错误及 HTTP 状态码 |
| 用户选择 Skill 但输入框为空 | 拒绝发送，提示「请输入内容」 |
| 活跃对话数达上限 | 提示归档旧对话（有子对话的父对话不计入配额） |
| 工具调用并发排队超时 | 丢弃未执行调用，注入系统提示 |
